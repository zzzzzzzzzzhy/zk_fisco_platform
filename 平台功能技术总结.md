# WEE Platform - Web3 内容分享与社区治理平台

## 📋 平台概述

WEE Platform 是一个基于 Polygon 区块链的 Web3 内容分享和 DAO 社区治理平台。平台通过代币激励机制，让创作者和平台方共同盈利，构建一个去中心化的内容生态系统。

**核心理念**：用户创作 → 获得代币奖励 → 参与社区治理 → 共享平台收益

### 关键业务流程（概览）

- **内容上传与存证流程**：
  - 前端上传图片 → 后端写入 MinIO（生成文件哈希） →  
    后端异步推送到 FISCO Guomi 链做企业级存证 → 同步/异步推送到 Polygon `ContentShareRegistry` 合约做 EIP-712 签名存证。
- **WEE 代币奖励流程**：
  - 发帖 / 评论 / 内容分享等业务成功 → 后端调用 Polygon `ForumTokenExtension` 合约对应的 `reward*` 函数 →  
    交易回执 `status=0x1` 视为成功，并记录到「WEE 代币账单」；`status=0x0` 前端显示失败原因链接到 Polygonscan。
- **容器启动流程**：
  - 在 `docker/` 目录执行 `docker compose up -d` → 启动 `backend / frontend / mysql / redis` 四个容器 →  
    后端容器通过环境变量连接容器内 MySQL / Redis，通过宿主机 IP 连接 MinIO 与 FISCO BCOS Guomi 节点。

## 💎 详细功能说明

### 一、代币奖励系统

#### 1.1 发帖奖励
- **奖励金额**：10 WEE / 帖子
- **触发条件**：成功发布帖子
- **实现方式**：后端调用智能合约 `rewardPost()`
- **防作弊**：每日上限 + 内容质量检测

#### 1.2 评论奖励
- **奖励金额**：5 WEE / 评论
- **触发条件**：成功发布评论
- **实现方式**：后端调用智能合约 `rewardComment()`
- **防作弊**：每日上限 + 内容长度验证

#### 1.3 每日签到
- **基础奖励**：5 WEE / 天
- **连续奖励**：
- **实现方式**：前端直接调用智能合约 `checkin()`
- **状态查询**：`canCheckinToday()` 检查是否可签到

#### 1.4 打赏功能
- **支持类型**：内容分享、论坛帖子
- **金额范围**：1-10000 WEE（可配置）
- **实现方式**：调用智能合约 `tipContent()`
- **防作弊**：禁止自我打赏
- **分成机制**：
  - 创作者：90%
  - 平台：10%（可配置）

#### 1.5 代币配置
```java
RewardConfig {
    postReward: 50 * 10^18 WEE
    commentReward: 20 * 10^18 WEE
    dailyReward: 100 * 10^18 WEE
    tipCreatorPercent: 90%
    tipPlatformPercent: 10%
    dailyPostRewardLimit: 500 * 10^18 WEE
    dailyCommentRewardLimit: 200 * 10^18 WEE
}
```

### 二、内容存证系统

#### 2.1 存证流程
1. **前端上传**：用户上传文件到 AWS S3
2. **生成哈希**：计算文件 SHA-256 哈希值
3. **EIP-712 签名**：后端生成结构化签名
4. **上链记录**：调用 `ContentShareRegistry.recordShare()`
5. **数据库存储**：保存交易哈希和元数据

#### 2.2 存证内容
```solidity
struct ShareRecord {
    bytes32 dataHash;        // 内容哈希
    address publisher;       // 发布者地址
    string shareId;          // 分享ID
    string metadata;         // 元数据（JSON）
    uint256 timestamp;       // 时间戳
    bool verified;           // 验证状态
}
```

#### 2.3 EIP-712 签名结构
```javascript
domain = {
    name: "ContentShareRegistry",
    version: "1",
    chainId: 137,
    verifyingContract: "0x477c1FC569eCefE56a4e3D54616CC83AFB7d02E3"
}

types = {
    ShareData: [
        { name: "dataHash", type: "bytes32" },
        { name: "publisher", type: "address" },
        { name: "shareId", type: "string" },
        { name: "metadata", type: "string" },
        { name: "timestamp", type: "uint256" }
    ]
}
```


```
   ↓
7. 刷新余额和价格
```

## 🏛️ DAO 治理与参数化设计（WEE 治理方案 v1）

> 目标：在现有 WEE 代币经济和内容激励体系之上，引入一套**轻量级、可演进的链上治理机制**，优先实现「奖励参数的链上治理」，在不中断现有业务的前提下，为未来完全 DAO 化留出空间。

### 1. 设计原则

- **渐进式去中心化**：优先把“可参数化的规则”上链，不一开始就把所有权限交给 DAO，而是从奖励参数、白名单等低风险配置做起。  
- **兼容现有业务**：优先改造 RewardConfig 读取逻辑，保持 `ForumTokenExtension` 的业务函数不变。  
- **WEE 单币双用**：v1 阶段使用同一个 WEE 代币承担「激励 + 治理」双重职能，降低心智/技术复杂度，后续可升级为 veWEE 或独立治理代币。  
- **自动执行**：治理提案一旦通过，由治理合约直接调用业务合约变更参数，而不是人工在后台改配置，保证**链上可验证**。  
- **安全边界清晰**：关键参数的变更范围、变更频率设有上限与延时（Timelock），避免恶意提案瞬间摧毁经济模型。  

---

### 2. 治理范围（v1 阶段）

首批纳入 DAO 治理的参数集中在 WEE 激励与经济相关的可配置项：

- **奖励类参数（来自 `ForumTokenExtension.RewardConfig`）**
  - `postReward`：发帖基础奖励（WEE）  
  - `commentReward`：评论基础奖励（WEE）  
  - `dailyCheckinReward`：每日签到奖励  
  - `consecutiveBonus`：连续签到奖励  
  - `contentImageReward`：内容分享-图片奖励  
  - `contentVideoReward`：内容分享-视频奖励  
- **限制类参数（可在 v2/v3 纳入）**
  - 每日发帖奖励上限（per-user / 全局）  
  - 每日评论奖励上限  
  - 单账号/单 IP 的上传速率限制（链下配置 + 链上开关）  
- **费用 / 手续费类（v2 阶段）**
  - 付费置顶价格 `_getPinPrice()`  
  - 打赏分成比例（创作者 / 平台）  
  - 平台从打赏/交易中抽成的最大上限  

> v1 只做 **RewardConfig 的链上治理**，其他高风险参数（如抽成比例、紧急暂停）仍由 Owner/多签掌握，系统稳定后再逐步上链。

---

### 3. 合约架构设计

#### 3.1 合约角色

- **WEE（MTK.sol）**  
  - 扩展为 `ERC20Votes`，支持链上投票权（1 WEE = 1 vote，后续可改为 veWEE）。  
  - 保持现有转账 / 打赏 / 交易逻辑不变。

- **ForumTokenExtension（激励合约）**  
  - 已存在 `RewardConfig` 结构；  
  - 新增 `setRewardConfig(RewardConfig calldata newConfig)` 函数，权限从 `onlyOwner` 迁移为 `onlyGovernance`；  
  - 其余业务函数（`rewardPost / rewardComment / rewardContentShare` 等）继续使用最新的 `rewardConfig`。

- **RewardGovernor（治理合约）**  
  - 基于 OpenZeppelin `Governor` + `GovernorCountingSimple` + `GovernorTimelockControl`；  
  - 治理 token 为 WEE（或未来的 veWEE）；  
  - 负责：
    - 创建提案（例如调整奖励参数）；  
    - 收集投票、计算通过与否；  
    - 通过后调用 `ForumTokenExtension.setRewardConfig()`。

#### 3.2 合约交互关系

```
WEE (ERC20Votes)  <——  用户质押 / 持有 → 拥有投票权
        │
        ▼
RewardGovernor (DAO)
        │  调用
        ▼
ForumTokenExtension.setRewardConfig(newConfig)
```

#### 3.3 示例：RewardConfig 结构与更新函数

```solidity
struct RewardConfig {
    uint256 postReward;
    uint256 commentReward;
    uint256 dailyCheckinReward;
    uint256 featuredPostReward;
    uint256 consecutiveBonus;
    uint256 contentImageReward;
    uint256 contentVideoReward;
}

contract ForumTokenExtension is Ownable {
    RewardConfig public rewardConfig;
    address public governance; // 治理合约地址

    modifier onlyGovernance() {
        require(msg.sender == governance || msg.sender == owner(),
            "ForumTokenExtension: not governance");
        _;
    }

    function setGovernance(address _gov) external onlyOwner {
        require(_gov != address(0), "invalid gov");
        governance = _gov;
    }

    function setRewardConfig(RewardConfig calldata cfg) external onlyGovernance {
        // 可选：增加参数边界检查，例如：reward 不超过某个上限
        rewardConfig = cfg;
    }
}
```

> 在 v1 阶段保留 `owner` 的兜底权限（紧急情况下可更改 `governance`），后续可将 Owner 转交给多签或完全交由治理合约。

---

### 4. 治理流程（提案 → 投票 → 执行）

#### 4.1 提案生命周期

1. **Proposal 创建**  
   - 发起人通过前端调用 `RewardGovernor.propose()`，携带：  
     - 调用目标：`target = address(ForumTokenExtension)`  
     - 调用数据：`calldata = abi.encodeWithSelector(setRewardConfig.selector, newConfig)`  
     - 描述：人类可读说明，例如 `将 postReward 从 10 WEE 提升到 15 WEE`。  

2. **投票期（Voting Period）**  
   - 持有 WEE 的用户通过 `castVote` / `castVoteWithReason` 投赞成/反对票；  
   - 投票权重 = 用户在该提案快照高度时的 `getVotes(account)`；  
   - 支持 `for / against / abstain` 三种投票选项。

3. **结果统计**  
   - 满足以下条件则视为通过（可在 Governor 中配置）：  
     - 得票率 ≥ 50%（或自定义 Quorum）；  
     - 赞成票 > 反对票；  
     - 参与票数 ≥ 最低投票门槛（例如至少 1% 流通 WEE 参与）。  

4. **执行（Execute）**  
   - 在投票通过且过了 Timelock 延时时，任何人都可调用 `execute()`；  
   - `RewardGovernor` 内部通过 `call()` 调用 `ForumTokenExtension.setRewardConfig(newConfig)`；  
   - 调用成功后，新的 RewardConfig 立即生效，后端/前端下次读合约就会得到最新参数。

#### 4.2 时序参数（示例）

- `votingDelay`：提案创建后延迟 1 个区块再开始投票（防止闪电贷）；  
- `votingPeriod`：例如 3 天（按区块高度换算）；  
- `timelockDelay`：执行前安全延时，例如 24 小时，用于社区观察与紧急响应。

### 5. 后端与前端集成方案

#### 5.1 后端：从链上读取 RewardConfig

在 `ForumTokenRewardService` 中新增 `fetchRewardConfig()`：

```java
public RewardConfig fetchRewardConfig() {
    var onChainCfg = forumTokenExtension.rewardConfig().send();
    RewardConfig cfg = new RewardConfig();
    cfg.setPostReward(onChainCfg.getPostReward());
    cfg.setCommentReward(onChainCfg.getCommentReward());
    cfg.setDailyCheckinReward(onChainCfg.getDailyCheckinReward());
    cfg.setFeaturedPostReward(onChainCfg.getFeaturedPostReward());
    cfg.setConsecutiveBonus(onChainCfg.getConsecutiveDays());
    cfg.setContentImageReward(onChainCfg.getContentImageReward());
    cfg.setContentVideoReward(onChainCfg.getContentVideoReward());
    return cfg;
}
```

- 现有的奖励逻辑（`rewardPost / rewardComment / rewardContentShare`）全部改为：  
  - **不再从 `application.yml` 读固定数值**，而是每次（或按一定间隔缓存）从链上 `rewardConfig` 读取。  

#### 5.2 前端：治理面板（Governance 页面）

**基础模块：**

- **参数总览卡片**  
  - 从合约读取当前 RewardConfig，并格式化展示：  
    - 发帖奖励 / 评论奖励 / 签到奖励 / 内容分享奖励等；  
    - 标注最近一次修改时间（可从合约 `RewardConfigUpdated` 事件解析）。  

- **提案列表**  
  - 通过 `RewardGovernor` 的事件（`ProposalCreated`、`VoteCast`、`ProposalExecuted`）拉取：  
    - 提案 ID、标题、发起人、创建时间、状态（Pending / Active / Succeeded / Defeated / Executed）；  
    - 投票详情：赞成票、反对票、参与地址数。

- **提案详情页**  
  - 展示提案变更前后的 RewardConfig 对比（diff）；  
  - 展示各地址的投票情况（可按票数排序）。  

- **操作入口**  
  - 发起提案：管理员或满足门槛的用户点击“新建提案”，填写新的参数和说明，前端构造 `propose()` 调用；  
  - 参与投票：已连接 MetaMask 的用户，前端调用 `castVote`/`castVoteWithReason`；  
  - 执行提案：当状态为 Succeeded 且过了锁定期，显示“执行”按钮，调用 `execute()`。  

> v1 阶段可以只对「管理员/白名单地址」开放发起提案入口，普通用户只开放投票，避免提案泛滥。

---

### 6. 安全与风控设计（治理相关）

- **参数上限/下限限制**  
  - 在 `setRewardConfig` 内部做强制约束，例如：  
    - 单项奖励上限不超过 `1000 WEE`；  
    - 每日总奖励上限不超过预设阈值；  
    - 抽成比例不得超过 50%。  

- **Timelock 延时执行**  
  - 所有会影响资金/奖励的变更都通过 `TimelockController` 执行，提供至少 24h 缓冲期；  
  - 出现恶意提案可在 Timelock 阶段由多签/Owner 触发 `cancel` 或紧急暂停。  

- **多签 + 治理分层（后续优化）**  
  - 合约 Owner 从单一 EOA 迁移到多签（如 Gnosis Safe），避免单点故障；  
  - DAO 主要管理“参数与资源配置”，技术运维（合约升级、紧急停机）由多签团队负责。
