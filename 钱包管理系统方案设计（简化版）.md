## 1. 业务需求分析

### 1.1 平台定位

**业务模式：**
- 本公司独立发布竞赛题目
- 用户报名参赛，提交作品
- 系统评测打分，生成排行榜
- 对获奖用户发放奖金（积分形式）
- 用户完成KYC认证后申请提现
- 平台线下银行转账支付
- 积分清零，完成闭环

**核心特点：**
- ❌ 不支持用户充值
- ❌ 不涉及在线支付
- ✅ 积分制钱包（虚拟货币）
- ✅ 银行转账提现（唯一方式）
- ✅ 强制KYC实名认证
- ✅ 区块链存证（透明公开）

### 1.2 核心业务场景

#### 场景1：奖金积分发放
- **触发条件**：竞赛结束，排行榜确认
- **业务流程**：
  1. 管理员创建奖金分配方案（按排名）
  2. 系统执行奖金批次发放
  3. 自动计算每个获奖用户的积分
  4. 风控审核通过
  5. 积分自动入账到用户钱包
  6. 发送入账通知（站内信+邮件）
  7. 区块链存证（批次Merkle Root上链）

#### 场景2：积分余额查询
- **触发条件**：用户查看钱包
- **展示内容**：
  - 可用积分（POINTS）
  - 冻结积分（提现申请中）
  - 历史收入积分
  - 已提现积分
  - 交易记录列表

#### 场景3：用户提现申请
- **触发条件**：用户主动申请提现
- **前置条件**：
  - ✅ 已完成KYC实名认证（L2级）
  - ✅ 积分余额≥最低提现金额（100积分）
  - ✅ 绑定银行卡
- **业务流程**：
  1. 用户填写提现信息
     - 提现积分数量
     - 收款银行卡信息（四要素验证）
     - 提现备注（可选）
  2. 系统验证
     - KYC状态检查
     - 积分余额检查
     - 银行卡信息验证
  3. 冻结相应积分
  4. 创建提现申请记录
  5. 记录交易流水（WITHDRAW_FREEZE）
  6. 上链存证（SUBMIT）
  7. 自动进入风控审核

#### 场景4：提现审核与打款
- **触发条件**：提现申请提交后
- **审核流程**：
  1. **自动风控审核**
     - 用户风险等级评估
     - 提现频率检查
     - 金额异常检查
     - 银行卡真实性验证
     - 生成风险评分
  2. **人工审核**（中高风险订单）
     - 管理员查看申请详情
     - 核实用户信息
     - 审核通过/拒绝
     - 填写审核意见
  3. **线下打款**
     - 财务人员登录网银
     - 手动转账到用户银行卡
     - 上传打款凭证（截图/回单）
     - 填写银行流水号
     - 确认打款完成
  4. **扣除积分**
     - 扣除冻结积分
     - 积分清零（提现金额对应的积分）
     - 记录交易流水（WITHDRAW_SUCCESS）
     - 上链存证（COMPLETE）
  5. **通知用户**
     - 站内消息通知
     - 邮件通知
     - 短信通知（可选）

### 1.3 积分与金额换算规则

**积分定义：**
- 1积分 = 1元人民币
- 积分为整数，不支持小数
- 积分只能增加（奖金）或减少（提现），不能转账

**提现规则：**
- 最低提现：100积分（100元）
- 最高单笔：50,000积分（5万元）
- 每日提现次数：3次
- 每月提现次数：30次
- 提现手续费：**免费**（平台承担转账手续费）

**积分有效期：**
- 永久有效，不过期
- 提现后对应积分清零
- 未提现积分保留在账户

## 3. 功能模块设计

### 3.1 积分钱包模块

#### 3.1.1 积分余额管理

**功能列表：**
- ✅ 查询用户积分余额（可用+冻结）
- ✅ 自动创建钱包（首次发放时）
- ✅ 增加积分（奖金发放）
- ✅ 冻结积分（提现申请）
- ✅ 解冻积分（提现失败/拒绝）
- ✅ 扣除积分（提现成功，积分清零）
- ✅ 乐观锁并发控制

**核心机制：**

##### 积分发放流程
```
竞赛奖金分配
    │
    ├─► 1. 管理员创建奖金批次
    │   └─► 指定竞赛、分配规则、总金额
    │
    ├─► 2. 系统计算各用户奖金
    │   └─► 按排名自动计算
    │
    ├─► 3. 批量发放积分
    │   ├─► 遍历获奖用户
    │   ├─► 增加积分余额
    │   ├─► 记录交易流水
    │   └─► 生成Merkle Root
    │
    ├─► 4. 区块链存证
    │   └─► 批次信息+Merkle Root上链
    │
    └─► 5. 通知用户
        ├─► 站内消息
        └─► 邮件通知
```

##### 积分清零规则
```
提现成功后：
- 扣除的积分 = 提现金额对应的积分
- 如：提现500元，扣除500积分
- 扣除后，该部分积分永久清零
- 剩余积分保留在账户
```

#### 3.1.2 交易流水管理

**交易类型（简化）：**

| 交易类型 | 代码 | 方向 | 说明 |
|---------|------|------|------|
| 积分发放 | PRIZE_IN | 入账 | 获奖后系统发放 |
| 提现冻结 | WITHDRAW_FREEZE | 冻结 | 提现申请时冻结 |
| 提现成功 | WITHDRAW_SUCCESS | 出账 | 打款成功，积分清零 |
| 提现拒绝 | WITHDRAW_REJECT | 解冻 | 审核拒绝，解冻积分 |
| 提现失败 | WITHDRAW_FAILED | 解冻 | 打款失败，解冻积分 |

**流水字段设计：**
```
id - 流水ID
user_id - 用户ID
type - 交易类型
amount - 交易积分数量
balance_before - 交易前积分
balance_after - 交易后积分
biz_ref - 业务关联ID（批次号/提现申请ID）
status - 状态（SUCCESS/FAILED）
remark - 备注
created_at - 创建时间
```

#### 3.1.3 对账模块

**日常对账：**
- **对账频率**：每日凌晨3点自动执行
- **对账公式**：
  ```
  当前总积分 = SUM(PRIZE_IN) - SUM(WITHDRAW_SUCCESS)

  验证：
  数据库余额总和 = 所有用户的 (可用积分 + 冻结积分)
  交易流水计算结果 = 当前总积分

  如果：数据库余额总和 == 交易流水计算结果
  则：对账成功 ✅
  否则：对账失败，生成差异报告 ❌
  ```

**异常处理：**
- 发现差异：立即告警（钉钉+邮件）
- 生成对账差异报告
- 人工介入核查
- 记录差异原因和处理结果

**对账报表：**
- 每日对账结果
- 每月对账汇总
- 年度对账报告

### 3.2 KYC实名认证模块

#### 3.2.1 认证等级设计

| 等级 | 名称 | 认证要求 | 提现限制 | 说明 |
|------|------|---------|---------|------|
| L0 | 未认证 | 无 | **不可提现** | 仅能接收积分 |
| L1 | 基础认证 | 姓名+身份证号 | 单笔1000，月限5000 | 快速认证 |
| L2 | 标准认证 | L1+人脸识别+银行卡四要素 | 单笔5万，无月限 | **推荐认证** |
| L3 | 高级认证 | L2+人工审核 | 单笔50万，无月限 | 大额用户 |

**推荐认证等级：**
- 普通用户：L2标准认证
- 大额用户：L3高级认证

#### 3.2.2 认证流程

```
用户点击提现
    │
    ├─► 检测KYC状态
    │   ├─► 未认证(L0)：引导到认证页面
    │   └─► 已认证：继续提现流程
    │
    ▼
【L1基础认证】
    ├─► 1. 填写基本信息
    │   ├─► 真实姓名
    │   └─► 身份证号
    │
    ├─► 2. 上传身份证照片
    │   ├─► 正面照片
    │   └─► 反面照片
    │
    ├─► 3. 自动OCR识别
    │   ├─► 提取身份证信息
    │   └─► 比对填写信息
    │
    ├─► 4. 调用公安部接口验证
    │   └─► 二要素验证（姓名+身份证号）
    │
    └─► 5. 认证结果
        ├─► 成功：升级到L1
        └─► 失败：提示重新提交
    │
    ▼
【L2标准认证（推荐）】
    ├─► 6. 人脸识别
    │   ├─► 调用活体检测
    │   ├─► 人脸与身份证照片比对
    │   └─► 防止盗用他人身份证
    │
    ├─► 7. 绑定银行卡
    │   ├─► 输入银行卡号
    │   ├─► 输入预留手机号
    │   └─► 四要素验证（姓名+身份证+银行卡+手机号）
    │
    └─► 8. 认证完成
        ├─► 成功：升级到L2，可正常提现
        └─► 失败：提示原因，重新验证
    │
    ▼
【L3高级认证（可选）】
    ├─► 9. 提交补充材料
    │   ├─► 手持身份证照片
    │   └─► 地址证明
    │
    ├─► 10. 人工审核
    │   ├─► 管理员查看资料
    │   └─► 审核通过/拒绝
    │
    └─► 11. 认证完成
        └─► 成功：升级到L3，大额提现
```

#### 3.2.3 第三方服务对接

**推荐服务：阿里云实人认证**

**接口说明：**
1. **OCR身份证识别**
   - API：`ocr_idcard`
   - 功能：识别身份证正反面信息
   - 价格：约0.05元/次

2. **二要素实名认证**
   - API：`id_card_verification`
   - 功能：验证姓名+身份证号真实性
   - 价格：约0.10元/次

3. **人脸识别活体检测**
   - API：`face_liveness`
   - 功能：防止照片/视频攻击
   - 价格：约0.30元/次

4. **银行卡四要素验证**
   - API：`bank_card_4element`
   - 功能：验证姓名+身份证+银行卡+手机号
   - 价格：约0.20元/次

**总成本：**
- L1认证：0.15元/人
- L2认证：0.65元/人
- L3认证：0.65元/人（+人工成本）

**接入步骤：**
1. 开通阿里云账号
2. 开通实人认证服务
3. 获取AccessKey
4. 集成SDK
5. 测试验证

#### 3.2.4 银行卡管理

**银行卡绑定：**
- 每个用户最多绑定3张银行卡
- 绑定时需四要素验证
- 默认第一张为提现卡
- 支持切换默认卡

**支持银行：**
- 工商银行、建设银行、农业银行、中国银行
- 招商银行、交通银行、邮储银行
- 民生银行、兴业银行、浦发银行
- 其他主流商业银行

**银行卡信息存储：**
- 银行卡号：加密存储（AES-256）
- 持卡人姓名：加密存储
- 预留手机号：加密存储
- 银行代码：明文存储
- 绑定时间：明文存储

#### 3.3.1 提现申请流程

用户发起提现

    │
    ├─► 2. 填写提现信息

    │   └─► 填写备注（可选）
    │
    ├─► 3. 确认提现
    │   ├─► 展示提现详情

    │   │   ├─► 手续费：0元（免费）
    │   │   ├─► 实际到账：XXX元
    │   │   └─► 收款银行卡：****1234
    │   └─► 用户确认提交
    │
    ├─► 4. 创建提现申请
    │   ├─► 生成申请ID
    │   ├─► 冻结积分
    │   ├─► 记录交易流水（WITHDRAW_FREEZE）
    │   ├─► 上链存证（SUBMIT）
    │   └─► 状态：PENDING（待审核）
    │
    ├─► 5. 自动风控审核
    │   ├─► 计算用户风险评分
    │   ├─► 执行风控规则引擎
    │   └─► 生成风险等级（LOW/MEDIUM/HIGH）

        └─► 高风险：转人工审核

```

    │
    ├─► 1. 管理员查看申请详情

    │
    ▼
【线下打款】
    │
    ├─► 5. 财务人员查看待打款列表
    │   └─► 按提交时间排序
    │
    ├─► 6. 批量导出打款清单
    │   ├─► 生成Excel文件
    │   │   ├─► 申请ID
    │   │   ├─► 用户姓名
    │   │   ├─► 银行卡号

    │   └─► 完成转账
    │
    ├─► 8. 上传打款凭证
    │   ├─► 截图转账成功页面
    │   ├─► 或上传电子回单
    │   └─► 系统保存到OSS
    │
    ├─► 9. 填写打款信息

#### 3.3.3 提现限制规则

**基础限制：**
- 最低提现：100积分（100元）
- 最高单笔：50,000积分（5万元）
- 每日次数：3次
- 每月次数：30次
- 手续费：**0元**（平台承担银行转账手续费）

**KYC等级限制：**
- L0未认证：不可提现
- L1基础认证：单笔≤1000积分，月限5000积分
- L2标准认证：单笔≤50000积分，无月限
- L3高级认证：单笔≤50万积分，无月限

**到账时间：**
- 工作日申请：T+1到账（次日到账）
- 非工作日申请：下个工作日开始处理，T+1到账
- 审核通过后：1-2个工作日到账

## 6. 银行转账方案

### 6.1 转账方式选择

**方案对比：**

| 方案 | 优势 | 劣势 | 成本 | 推荐度 |
|------|------|------|------|--------|
| **企业网银手动转账** | 操作简单，无技术成本 | 效率低，易出错 | 手续费约2-5元/笔 | ⭐⭐⭐⭐⭐ |
| 银企直连API | 自动化程度高 | 开发复杂，需企业网银U盾 | 开发成本高 | ⭐⭐⭐ |
| 第三方支付平台 | 接入简单 | 费率较高（0.3%） | 按比例收费 | ⭐⭐⭐⭐ |

**推荐方案：企业网银手动转账**

**理由：**
- ✅ 技术实现简单，无需对接复杂API
- ✅ 资金安全，直接控制
- ✅ 手续费固定，成本可控
- ✅ 适合初期用户量不大的情况
- ❌ 效率相对较低（可通过批量转账提升）

### 6.2 转账流程设计

#### 6.2.1 批量打款流程

```
【财务人员操作】

1. 登录管理后台
   └─► 进入"提现管理"模块

2. 筛选待打款订单
   ├─► 状态：APPROVED（审核通过）
   ├─► 时间范围：今天/本周
   └─► 按创建时间升序排列

3. 批量导出打款清单
   ├─► 点击"导出Excel"按钮
   ├─► 系统生成Excel文件
   │   ├─► 包含：姓名、银行卡号、金额、开户行
   │   └─► 按银行分组（可选）
   └─► 下载到本地

4. 登录企业网银
   ├─► 选择"批量转账"功能
   ├─► 导入Excel文件
   │   └─► 网银解析转账清单
   ├─► 核对转账信息
   ├─► 插入U盾，输入密码
   └─► 提交批量转账

5. 转账执行
   ├─► 银行处理转账
   ├─► 实时/T+0到账
   └─► 生成电子回单

6. 上传转账凭证
   ├─► 截图转账成功页面
   ├─► 或下载电子回单PDF
   ├─► 回到管理后台
   ├─► 逐笔或批量上传凭证
   └─► 填写银行流水号

7. 确认打款完成
   ├─► 系统自动扣除积分
   ├─► 更新提现状态：COMPLETED
   ├─► 发送通知给用户
   └─► 区块链存证
```

#### 6.2.2 单笔打款流程（小额）

```
1. 管理员查看提现申请详情
2. 手动登录网银
3. 选择"单笔转账"

   - 收款人姓名
   - 银行卡号

### 6.3 银行选择建议

**推荐银行：**

**1. 中国工商银行**
- 优势：网点最多，手续费较低
- 批量转账：支持，最多1000笔/批
- 手续费：2元/笔（同行），5元/笔（跨行）
- 到账时间：实时（同行），2小时内（跨行）

**2. 中国建设银行**
- 优势：企业网银功能强大
- 批量转账：支持，最多2000笔/批
- 手续费：1元/笔（同行），5元/笔（跨行）
- 到账时间：实时（同行），T+0（跨行）

**3. 招商银行**
- 优势：网银体验好，转账快
- 批量转账：支持，最多500笔/批
- 手续费：免费（同行），5元/笔（跨行）
- 到账时间：实时

**建议：**
- 开通2-3家银行的企业账户
- 分散风险，提高效率
- 优先使用手续费低的银行

### 6.4 转账异常处理

**常见异常：**

**1. 银行卡信息错误**
- 提示：收款人姓名与银行卡不符
- 处理：
  - 联系用户确认信息
  - 用户修改银行卡信息
  - 重新发起转账

**2. 银行卡状态异常**
- 提示：该卡已冻结/注销
- 处理：
  - 联系用户更换银行卡
  - 重新绑定
  - 重新发起转账

**3. 转账限额**
- 提示：超过单笔限额
- 处理：
  - 分拆为多笔转账
  - 或升级企业网银限额

**4. 网银故障**
- 提示：网银系统维护
- 处理：
  - 等待系统恢复
  - 或切换其他银行

### 6.5 手续费管理

**手续费承担：**
- 平台承担所有转账手续费
- 用户提现0手续费

**成本计算：**
```
假设：
- 平均每笔提现金额：1000元
- 跨行转账手续费：5元/笔
- 每月提现笔数：1000笔

每月成本 = 1000 × 5 = 5000元/月
年度成本 = 5000 × 12 = 60000元/年
```

**优化建议：**
- 引导用户绑定平台主营银行（降低跨行比例）
- 协商银行手续费折扣
- 定期对账，避免重复转账

## 7. KYC认证方案

### 7.1 认证流程设计

#### 7.1.1 L1基础认证

**认证步骤：**
1. 用户填写姓名+身份证号
2. 上传身份证正反面照片
3. 系统OCR识别身份证信息
4. 调用公安部二要素验证接口
5. 验证通过，升级到L1

**所需时间：**约3-5分钟

**通过率：**约95%

**失败原因：**
- 身份证号错误
- 照片不清晰
- 身份证过期
- 公安部接口超时

#### 7.1.2 L2标准认证（推荐）

**认证步骤：**
1. 完成L1基础认证
2. 进行人脸识别
   - 调用活体检测（防止照片攻击）
   - 人脸与身份证照片比对
3. 绑定银行卡
   - 输入银行卡号、预留手机号
   - 调用四要素验证接口
4. 验证全部通过，升级到L2

**所需时间：**约10-15分钟

**通过率：**约85%

**失败原因：**
- 人脸识别不通过（相似度<80%）
- 银行卡四要素验证失败
- 手机号错误

#### 7.1.3 L3高级认证（可选）

**认证步骤：**
1. 完成L2标准认证
2. 上传手持身份证照片
3. 上传地址证明（水电费账单等）
4. 提交给人工审核
5. 管理员审核通过，升级到L3

**所需时间：**1-2个工作日

**适用场景：**
- 单笔提现>5万
- 月提现>10万
- 企业用户

## 8. 安全方案

#### 8.1.2 存储加密

**敏感字段加密：**
```java
// 加密算法：AES-256-GCM
public class AESUtil {
    private static final String SECRET_KEY = System.getenv("AES_SECRET_KEY");

    public static String encrypt(String plaintext) {
        // AES-256-GCM加密

    }

        // AES-256-GCM解密

    }

```

**密钥管理：**
- 密钥存储在环境变量
- 定期轮换（每季度）
- 多环境隔离（开发/测试/生产）

log.info("用户{}提现{}积分，银行卡{}",

| 提现频率 | 15% | 异常频繁:-50分；正常:100分 | -50-100 |
| 金额合理性 | 10% | 超过历史平均3倍:-30分；正常:100分 | -30-100 |
| 行为异常 | 10% | 有异常:-50分；无异常:100分 | -50-100 |

**总分计算：**
```
总分 = 账户年龄×0.2 + KYC等级×0.25 + 历史提现×0.2
     + 提现频率×0.15 + 金额合理性×0.1 + 行为异常×0.1
```

**风险等级划分：**
- 0-40分：高风险（HIGH）→ 人工审核
- 41-70分：中风险（MEDIUM）→ 延迟审核
- 71-100分：低风险（LOW）→ 自动通过

### 9.2 风控规则引擎

**规则列表：**

**规则1：新用户首次提现**
```
条件：注册时间<7天 AND 首次提现
风险等级：MEDIUM
处理：转人工审核
```

**规则2：频繁提现**
```
条件：24小时内提现次数>3次
风险等级：HIGH
处理：拒绝 + 账户冻结24小时
```

**规则3：大额提现**
```
条件：单笔金额>10000积分
风险等级：MEDIUM
处理：转人工审核
```

**规则4：异地登录提现**
```
条件：登录IP城市变化 AND 30分钟内提现
风险等级：HIGH
处理：要求二次验证（短信验证码）
```

**规则5：未认证或低等级认证**
```
条件：KYC等级=L0 OR (L1 AND 金额>1000)
风险等级：HIGH
处理：拒绝 + 提示完成认证
```

**规则6：银行卡频繁更换**
```
条件：7天内更换银行卡>2次 AND 立即提现
风险等级：HIGH
处理：转人工审核
```

**规则7：夜间提现**
```
条件：提现时间在23:00-7:00
风险等级：MEDIUM
处理：延迟到工作时间审核
```

### 9.3 风控流程

```
提现申请提交
    │
    ├─► 1. 基础校验
    │   ├─► KYC状态（必须≥L1）
    │   ├─► 积分余额（≥提现金额）
    │   ├─► 银行卡状态（已验证）
    │   └─► 提现次数（未超限）
    │
    ├─► 2. 计算用户风险评分
    │   ├─► 账户年龄：30天 → 50分
    │   ├─► KYC等级：L2 → 90分
    │   ├─► 历史提现：2次 → 60分
    │   ├─► 提现频率：正常 → 100分
    │   ├─► 金额合理性：正常 → 100分
    │   ├─► 行为异常：无 → 100分
    │   └─► 总分 = 79分
    │
    ├─► 3. 执行风控规则引擎
    │   ├─► 规则1：不触发（账户>7天）
    │   ├─► 规则2：不触发（24小时内首次）
    │   ├─► 规则3：不触发（金额1000<10000）
    │   ├─► 规则4：不触发（同城登录）
    │   ├─► 规则5：不触发（L2认证）
    │   ├─► 规则6：不触发（银行卡未变更）
    │   └─► 规则7：不触发（白天提现）
    │
    ├─► 4. 生成风险评估报告
    │   ├─► 风险等级：LOW（79分）
    │   ├─► 命中规则：无
    │   └─► 建议处理：自动通过
    │
    └─► 5. 自动决策
        ├─► 低风险：状态更新为APPROVED，进入打款队列
        ├─► 中风险：状态更新为PENDING，4小时后自动转APPROVED
        └─► 高风险：状态更新为PENDING，转人工审核
```

### 9.4 黑名单机制

**黑名单类型：**

| 类型 | 触发条件 | 处理措施 | 解除方式 |
|------|---------|---------|---------|
| 用户黑名单 | 欺诈行为确认 | 禁止提现，冻结账户 | 人工解除 |
| IP黑名单 | 异常IP攻击 | 限制访问 | 24小时自动解除 |
| 设备黑名单 | 设备指纹异常 | 增强验证 | 7天自动解除 |
| 银行卡黑名单 | 可疑卡号 | 拒绝绑定/提现 | 人工审核解除 |

**黑名单数据来源：**
- 内部发现：风控系统标记
- 人工标记：审核人员标记
- 外部来源：同业共享、监管通报

## 10. 区块链集成方案

### 10.1 上链存证设计

**上链时机：**

| 业务场景 | 上链时机 | 上链数据 |
|---------|---------|---------|
| 积分发放 | 批次执行完成后 | 批次号 + Merkle Root |
| 提现申请 | 申请提交时 | 申请ID + 数据哈希 |
| 提现审核 | 审核完成时 | 申请ID + 审核结果 + 审核人 |
| 提现完成 | 打款确认后 | 申请ID + 打款凭证哈希 |

**数据哈希计算：**
```java
// 提现申请哈希
String dataHash = SHA256(
    userId + "|" +
    amount + "|" +
    bankCardNo + "|" +
    timestamp
);

// 打款凭证哈希
String proofHash = SHA256(proofFileBytes);
```

### 10.2 Merkle Tree防篡改

**积分发放批次上链：**
```
1. 获取批次所有发放记录
   User1: 5000积分
   User2: 3000积分
   User3: 2000积分
   ...

2. 为每个用户生成哈希
   Hash1 = SHA256("User1|5000|BATCH_001")
   Hash2 = SHA256("User2|3000|BATCH_001")
   Hash3 = SHA256("User3|2000|BATCH_001")
   ...

3. 构建Merkle Tree
                  Root
                /      \
           Hash12      Hash34
           /    \      /    \
       Hash1  Hash2  Hash3  Hash4

4. 上链存证
   bizType: "PRIZE_BATCH"
   bizId: "BATCH_001"
   dataHash: MerkleRoot
   metadata: {
     "竞赛ID": 10001,
     "总金额": 50000,
     "用户数": 100,
     "批次号": "BATCH_001"
   }

5. 防篡改验证
   - 任何一个用户的发放记录被修改
   - 重新计算Merkle Root
   - 与链上Root不一致 → 检测到篡改
```

### 10.3 链上数据查询

**用户查询：**
- 查看自己的积分发放记录
- 查看上链时间、交易哈希
- 验证数据真实性

**公开查询：**
- 任何人都可以验证数据
- 输入数据哈希，查询链上记录
- 确认数据未被篡改

**接口示例：**
```
GET /api/blockchain/verify/{dataHash}

```
