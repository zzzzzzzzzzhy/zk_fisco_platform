## 隐私充值 / 提现 / 募资：私密池 + ZK 设计草案（初版）

> 目标：在不影响现有透明业务（发帖、点赞、打赏、WEE 奖励等）的前提下，为「法币 ↔ 链上资产」这一层增加隐私与审计能力。  
> 思路：**普通业务继续走公开链；涉及资金进出、募资等敏感操作，走一个独立的“私密池 + 零知识证明”子系统。**

---

### 1. 总体思路（为什么要拆成子系统）

- 当前平台的主业务：
  - 内容发帖 / 评论 / 置顶；
  - WEE 奖励、打赏、签到等链上操作（Polygon 主网，ForumTokenExtension 合约）。
- 「隐私需求」主要集中在：
  - 用户从法币世界 **充值** 到平台（例如买 WEE 或参与募资）；
  - 用户从平台 **提现** 回法币世界；
  - 有些 **募资 / 投票 / 捐赠** 场景，希望金额和身份不对外公开。
- 因此：
  - **不需要把现有所有合约都变成 ZK**（复杂度和成本太高）；
  - 只需要在**法币 ↔ 链上资产这一层**，引入一个「私密池（Private Pool）+ 零知识证明」的模块，负责：
    - 对外：和法币支付渠道 / 银行对接；
    - 对内：发行“私密余额”，并维护一个隐私账本；
    - 用 zk 证明：用户在不暴露具体地址 / 金额的情况下，仍然能证明「我有足够余额」「这次操作合法」。

---

### 2. 模块划分（系统分层）

**2.1 现有公开层（保持不变）**

- Polygon 主网：
  - `ForumToken` / `ForumTokenExtension`：WEE 代币与奖励逻辑；
  - `ContentShareRegistry`：内容存证合约；
  - 这些合约继续采用公开地址 + 透明交易，方便审计和社区监督。
- 后端：
  - 继续负责奖励结算、Gas 监控、内容上链验证等；
  - 不引入 ZK 复杂度。

**2.2 新增「私密金融层」：Private Pool + ZK**

独立出来一个子系统（可以是一个单独的服务 + 一组合约）：

- **私密池合约（Private Pool Contract）**：
  - 接收来自公开地址的 Token（如 WEE 或 USDC）；
  - 在合约内部记录一份“承诺(commitment)”列表，而不是直接暴露每个用户的余额；
  - 支持：`deposit` / `withdraw` / （可选）`shieldedTransfer` 等操作。
- **ZK 电路 / 证明服务（Prover Service）**：
  - 封装 zk-SNARK / zk-STARK 等证明逻辑；
  - 为前端 / 后端提供生成和验证证明的 API；
  - 合约端只负责验证 proof 是否有效。
- **合规 & 审计服务（Off-chain）**：
  - 在数据库中维护 `user_id ↔ KYC 信息 ↔ 私密池身份` 的映射（只在后端可见）；
  - 保存每次充值 / 提现的 off-chain 审计日志（时间、法币金额、订单号、链上 commitment 等）。

**2.3 前端集成**

- 对于普通操作（发帖、点赞、奖励）：
  - 前端继续走原有流程，使用 MetaMask、公开地址。
- 对于充值 / 提现 / 私密募资：
  - 前端引导用户进入一个「隐私钱包 / 私密资产」页面；
  - 前端负责：
    - 生成随机密钥 / note；
    - 调用后端 ZK Prover，生成零知识证明；
    - 调用私密池合约，提交 `commitment` + `proof`。

---

### 3. 关键流程示例

#### 3.1 充值（法币 → 私密资产）

1. **用户在平台发起充值请求**（如：充值 100 USDT，或购买 1000 WEE 的“私密余额”）：
   - 前端：选择金额与支付渠道（支付宝/银行卡/第三方支付）；
   - 后端：生成一个充值订单（order_id），等待法币到账。
2. **法币到账（Off-chain）**：
   - 支付回调 / 对账确认后，后端标记订单为 `PAID`。
3. **生成私密资产（链上）**：
   - 后端为该用户生成一个 `commitment`（例如 Pedersen 承诺或 Poseidon 承诺）；
   - 在本地记录：`user_id ↔ commitment ↔ amount`；
   - 在链上私密池合约中执行 `deposit(order_id, commitment, amount)`：
     - 可能由平台统一地址代币化，例如：
       - 平台先从自己的公开地址把 1000 WEE 存入私密池；
       - 合约中增加一个新的承诺节点；
     - 或者如果用户本身已经有 WEE，也可以由用户自己调用 `deposit`。
4. **返回一个“私密凭证（note）”给用户**：
   - 前端展示并提示用户安全保存（可以加密存本地 / 手机钱包）；
   - note 中包含：`commitment` 的随机数、金额等信息，后续提现会用到。

> 效果：链上只看到「平台地址向私密池合约转入 1000 WEE，生成一个新的 commitment」，看不到是哪位用户、也不知道后面会怎么分配。

#### 3.2 平台内使用：公开业务 / 私密业务的结合

**公开业务继续用现在的方式**：

- 发帖奖励 / 评论奖励 / 签到 / 置顶等：
  - 仍然是公开地址 + 公开 ForumTokenExtension 合约；
  - 好处：规则透明、便于社区审核。

**私密用途（可选扩展）**：

- 如果未来需要搞「私密募资 / 私密赞助」：
  - 用户可以选择从「私密池余额」中参与募资；
  - 平台只在私密池里记录“某个 commitment 向某个项目 commitment 转移了多少金额”；
  - 对外只看到某个项目的「总募集金额」，看不到每个用户的贡献。

#### 3.3 提现（私密资产 → 法币）

1. **用户提交提现请求**：
   - 在前端输入：提现金额 + 接收账户（银行卡 / USDT 地址等）；
   - 前端从用户的私密 note 中，构造「我要花掉某个 commitment」的证明请求。
2. **生成零知识证明（前端 + Prover 服务）**：
   - 前端调用 ZK Prover（后端服务），传入：
     - 用户的 note（承诺随机数等）；
     - 预期提现金额；
     - 当前池子状态的 Merkle 根（由后端/合约提供）。
   - Prover 输出：
     - 一个 zk proof，证明：
       - 这个 commitment 存在于当前 Merkle 树中；
       - 该 commitment 尚未被花费（防止双花）；
       - 提现金额 <= commitment 的余额。
3. **向私密池合约提交提现交易**：
   - 前端或后端发起链上调用 `withdraw(proof, publicInputs)`：
     - 合约验证证明；
     - 记录该 commitment 已花费；
     - 将对应的 token 从私密池发送到平台的“出金地址”或用户的公开地址（具体取决于设计）。
4. **线下打款 / 出金**：
   - 如果是法币提现：平台根据提现订单，从企业账户转账到用户银行卡；
   - 如果是链上提现（比如直接提现成 USDT）：
     - 可以直接从私密池把 Token 打到用户指定地址（这个地址可以是新地址，提高隐私）。

> 效果：链上只看到「某个匿名 commitment 被花费了一次，池子里少了相应 amount」，看不到是谁在提现、具体走到哪里；平台内部则通过订单与 KYC 记录对应关系。

### 4. 跟现有系统如何对接

**4.1 不动现有合约和业务逻辑**

- `ForumTokenExtension` / `ContentShareRegistry` 等仍然是公开合约；
- 现有 Polygon 主网交互（MetaMask 签名、Gas 监控、WEE 账单）全部保留；
- 新增的私密池合约只在「充值 / 提现 / 私密募资」时用到。

**4.2 后端对接点**

- 新增一个独立的 Spring 模块 / 服务，例如：
  - `zk-privacy-service`：
    - 封装调用 ZK Prover（可以是本地进程或外部服务）；
    - 封装和私密池合约的交互（类似现在的 `ForumTokenRewardService`）。
- 数据库新增几张表（示意）：
  - `private_deposits`：记录每笔法币充值与 commitment 的映射；
  - `private_withdrawals`：记录每笔提现请求、证明哈希、状态；
  - `private_commitments`：可选，镜像链上的 commitment 列表，加速查询和风控。

**4.3 前端对接点**

- 新增一个「隐私资产」页面：
  - 展示：私密余额（只在后端可见，前端通过 API 拿到聚合后的数字）；
  - 功能：发起充值、提现、参与私密募资；
  - 在发起链上操作时：
    - 使用 MetaMask 与私密池合约交互；
    - 使用后端 Prover API 生成零知识证明。

### 5. 技术选型与实现难度（现实建议）

**5.1 可选技术栈**

- ZK 证明：
  - zk-SNARK（Groth16/Plonk）：证明体积小、验证快，但需要可信设置（trusted setup）；
  - zk-STARK：不需要 trusted setup，但证明更大、Gas 成本更高；
  - 现成组件：Circom + snarkjs、Halo2、Plonky2 等。
- 私密池合约：
  - 可以参考现有设计：
    - Tornado Cash 的匿名池模型（需注意合规）；
    - Aztec、Railgun 等开源方案（更复杂，但更通用）。

**5.2 难度与迭代建议**

- 难点主要在：
  - 电路设计与正确性（防止资金丢失 / 被盗）；
  - 性能（证明生成时间、链上验证 Gas 成本）；
  - 审计与合规（尤其在中国 / 企业内部环境）。
- 建议路线：
  1. **第一阶段**：仅在**产品层面**划分清楚——透明层 vs 私密层；
  2. **第二阶段**：先做一个**最小可用版本**：
     - 小额、内部测试用的私密充值 / 提现流程；
     - 使用简单的现成 ZK 模板（例如“匿名 set membership + 防双花”）。
  3. **第三阶段**：根据实际需求，逐步把更多募资 / 捐赠场景搬到私密池上。

---

### 6. 现阶段对你项目的建议

- **现在可以先不落地 ZK，只把架构预留出来**：
  - 文档里明确：未来所有「法币 ↔ 链上」和「需要强隐私的募资」会走一个独立的“私密金融层”；
  - 现有 WEE 奖励和内容存证继续用公开合约，保证简单 + 透明。
- 等项目进入「需要对外募资 / 大额资金管理」阶段，再：
  - 选择 1–2 个 ZK 方案做 PoC；
  - 把上面描述的「私密池 + ZK」子系统作为一个独立模块接入主平台。

> 这样设计的好处：  
> - **现在不背负复杂度**，主功能可以快速上线；  
> - 将来如果要上隐私和合规，只需要在「充值/提现/募资」边界增加这个子系统，不会推翻现有代码。
